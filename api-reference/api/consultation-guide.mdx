<Info>
**Authentication:** All endpoints below require `Authorization: Bearer <access_token>`.
**Self-scope:** Path `user_id` must match the JWT subject — cross-user access is rejected
with `403`.
**Inbound webhook:** `POST /webhooks/consultations/narayana/{benefit_id}` is **public** —
documented separately at the end of this page.
</Info>

## Endpoints at a Glance

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/users/{user_id}/benefits/{benefit_id}/eligible_dependants` | List which dependants the user can open consultations for, under this specific consultation benefit |
| `POST` | `/users/{user_id}/consultations` | Open a new consultation thread |
| `GET` | `/users/{user_id}/consultations` | Paginated list of the user's consultations |
| `GET` | `/users/{user_id}/consultations/{consultation_id}` | Detail (with live `unread_count`) |
| `GET` | `/consultations` | **Admin** — cross-user list (trusted backend only) |
| `GET` | `/consultations/{consultation_id}` | **Admin** — fetch any consultation by id |
| `GET` | `/users/{user_id}/consultations/{consultation_id}/messages` | Paginated message list |
| `POST` | `/users/{user_id}/consultations/{consultation_id}/messages` | Send a message (multipart) |
| `POST` | `/users/{user_id}/consultations/{consultation_id}/messages/{chat_message_id}/read_receipt` | Mark a message read |
| `GET` | `/users/{user_id}/consultations/{consultation_id}/messages/{chat_message_id}/attachment` | Download an attachment (binary stream) |

## 1. Discover Eligible Dependants

```bash
GET /users/{user_id}/benefits/{benefit_id}/eligible_dependants
```

Scoped by the consultation `benefit_id` in the URL path — caller asks "for this
specific consultation benefit, which of my dependants are eligible?". Returns
`{ dependants: [...] }`. Each entry includes `dependant_id`, name fields,
`insurance_policy_id`, and `active_consultation_count`. (The `benefit_id` is in
the URL, so it's not repeated per item.)

**What happens server-side:** Validates the benefit exists and is of type
`Consultation` (400 `CN-1615` otherwise). Loads the user's `Issued` policies,
filters to those whose provider also offers this benefit, and for each matched
`(dependant, benefit)` returns the active consultation count.

<CodeGroup>
```bash curl
curl -X GET https://api.aarokya.in/users/{user_id}/benefits/{benefit_id}/eligible_dependants \
  -H "Authorization: Bearer $TOKEN"
```

```json 200 OK
{
  "dependants": [
    {
      "dependant_id": "019e0f3f-...",
      "first_name": "Jane",
      "last_name": "Doe",
      "salutation": "MS",
      "gender": "FEMALE",
      "relationship": "SPOUSE",
      "insurance_policy_id": "019e0f3f-...",
      "active_consultation_count": 0
    }
  ]
}
```
</CodeGroup>

## 2. Open a Consultation

```bash
POST /users/{user_id}/consultations
```

<ParamField body="dependant_id" type="UUID" required>
The dependant the consultation is for. Must be eligible (see above).
</ParamField>

<ParamField body="benefit_id" type="UUID" required>
The consultation benefit returned in `eligible_dependants`.
</ParamField>

**What happens server-side:**

1. Validate eligibility for `(user_id, dependant_id, benefit_id)`.
2. Resolve the provider context (Narayana today).
3. Auto-register the dependant on Narayana if they don't yet have an MRN.
4. Create the upstream conversation, capture `external_reference_id`.
5. Insert a mirror row with `status = active`.
6. Return the response with `unread_count` from the upstream create.

A dependant may hold multiple concurrent active consultations under the same benefit —
there is no active-consultation uniqueness constraint.

<ResponseField name="status" type="string">
`201 Created` on success.
</ResponseField>

## 3. List Consultations

```bash
GET /users/{user_id}/consultations?dependant_id=...&benefit_id=...&statuses=ACTIVE,COMPLETED&limit=20&offset=0
```

All query params are optional. The list is sourced from the DB; per-row `unread_count`
is enriched best-effort via a single upstream `list_threads` call.

---

## 4. Get a Single Consultation

```bash
GET /users/{user_id}/consultations/{consultation_id}
```

Performs a best-effort upstream `get_thread` to populate `unread_count` and lazily
mirror status drift (`Active → Completed`/`Closed`). On upstream failure the mirror
row is returned unchanged with no `unread_count`, and a `warn!` line is logged
(`tag = "ConsultationGetUpstream"`).

---

## 4a. Admin (cross-user) — List & Get

```bash
GET /consultations?primary_user_id=...&dependant_id=...&benefit_id=...&statuses=ACTIVE,COMPLETED&limit=20&offset=0
GET /consultations/{consultation_id}
```

<Info>
**Trusted backend only** — these require an admin/operator (trusted-backend) token, not the
user's bearer token. Unlike the user-scoped routes there is no `{user_id}` in the path; all
filters are optional, so omitting them lists across every user.
</Info>

Both are **DB-only** — they do not call the upstream provider, so `unread_count` is always
absent here. To view a consultation's message thread from an admin context, take the
`external_reference_id` from the row and use the admin Narayana endpoints
(`/admin/narayana/conversations/{thread_id}/messages`).

## 5. List Messages

```bash
GET /users/{user_id}/consultations/{consultation_id}/messages?limit=50&offset=0
```

Aarokya exposes `limit` / `offset` for consistency with the rest of the platform.
Internally Narayana paginates by zero-indexed `page` of fixed `size`; the conversion is
`size = limit`, `page = offset / limit`.

<Info>
  The response is a **uniform envelope** (`total` / `limit` / `offset`) whose `data` field
  is `ConsultationMessages { provider, messages }`. The `provider` field is informational
  (it carries the upstream that backed this thread — today only `NARAYANA`); the
  `messages` array is a **neutral shape we own** regardless of provider. Every
  `ConsultationMessage` carries `id`, `sender_display_name`, `sender_role`, `created_at`
  (UTC ISO 8601), and a discriminated `body`. The `provider` tag appears on the `data`
  object even when `messages` is empty.

  `sender_role` is one of `PATIENT | DOCTOR | SYSTEM | UNKNOWN` — classified server-side
  from upstream sender metadata so UIs can render alignment without parsing display names.
</Info>

Each message `body` is a tagged union — exactly one of `text`, `attachment`, or
`system_event` — with our owned `SystemEventKind` vocabulary
(`PARTICIPANT_ADDED | PARTICIPANT_REMOVED | TOPIC_UPDATED | OTHER`):

```json
{
  "total": 3,
  "limit": 50,
  "offset": 0,
  "data": {
    "provider": "NARAYANA",
    "messages": [
      {
        "id": "19:acsV2_msg_...",
        "sender_display_name": "Dr. Jane",
        "sender_role": "DOCTOR",
        "created_at": "2026-04-28T16:00:00Z",
        "body": { "message_type": "TEXT", "content": "How are you feeling today?" }
      },
      {
        "id": "19:acsV2_msg_...",
        "sender_display_name": "Patient:Alice",
        "sender_role": "PATIENT",
        "created_at": "2026-04-28T16:02:00Z",
        "body": {
          "message_type": "ATTACHMENT",
          "id": "19:acsV2_msg_...",
          "file_name": "lab-report.pdf",
          "mime_type": "application/pdf",
          "file_size": 184512
        }
      },
      {
        "id": "19:acsV2_sysevt_...",
        "sender_display_name": "",
        "sender_role": "SYSTEM",
        "created_at": "2026-04-28T15:59:50Z",
        "body": { "message_type": "SYSTEM_EVENT", "kind": "PARTICIPANT_ADDED" }
      }
    ]
  }
}
```

<Note>
  A future provider (e.g. Apollo) reuses this exact wire shape — the only change is the
  `provider` tag value (e.g. `"APOLLO"`); existing parsers keep working. See
  [ADR-005](/decisions/adr-005-neutral-consultation-message-shape) (supersedes
  [ADR-004](/decisions/adr-004-provider-keyed-consultation-messages)).
</Note>

## 6. Send a Message (Multipart)

```bash
POST /users/{user_id}/consultations/{consultation_id}/messages
Content-Type: multipart/form-data
```

| Part | Type | Required | Notes |
|---|---|---|---|
| `content` | text part | when no file | Required if no `files` part is sent |
| `files` | binary file | when no text | **At most one** — a second `files` part returns `400` (CN-1612) |

At least one of `content` or `files` must be present. Narayana accepts a single
attachment per message, so only one `files` part is allowed. Multipart is hand-written
(Smithy doesn't model multipart well) — the request shape will not appear in the
generated SDK types.

---

## 7. Read Receipt

```bash
POST /users/{user_id}/consultations/{consultation_id}/messages/{chat_message_id}/read_receipt
```

Idempotent — duplicate sends return 200. The `chat_message_id` is the upstream
message id surfaced in `list_messages` responses.

---

## 8. Download Attachment

```bash
GET /users/{user_id}/consultations/{consultation_id}/messages/{chat_message_id}/attachment
```

Streams the binary attachment for an `attachment` message. Upstream `Content-Type`
and `Content-Disposition` headers are forwarded verbatim.

## Inbound Webhook (Public)

```bash
POST /webhooks/consultations/narayana/{benefit_id}
```

<Warning>
Public — **no Bearer token**. Narayana does not currently sign webhook requests.
Trust relies on URL scoping, payload shape, and IP allow-listing at the edge.
</Warning>

`{benefit_id}` scopes the callback to the consultation benefit it was registered
against. The handler:

1. Looks up the consultation by upstream `chat_conversation_source_id`.
2. Cross-checks that the row's `benefit_id` matches the URL.
3. Logs `tag = "NarayanaWebhook"` with `benefit_matches`, `recognised`, and the
   consultation id (no PHI).
4. Always returns 200 — Narayana re-fires on non-2xx responses, which we don't want
   for unknown threads or transient DB hiccups.

```json
{
  "id": "19:acsV2_msg_abc123",
  "chatConversationSourceId": "19:acsV2_thread_abc@thread.v2",
  "type": "text",
  "createdOn": "2026-04-28T10:30:00Z"
}
```

## Validation Rules

| Field | Rule |
|---|---|
| `dependant_id`, `benefit_id`, `consultation_id`, `chat_message_id` | UUID v7 format. Bad UUIDs → 400 |
| Path `user_id` | Must equal JWT subject (or trusted-backend actor). Mismatch → 403 |
| Send message: `content` | Required when no file. Trimmed; empty → 400 |
| Send message: `files` | At most one part — a second → 400 (CN-1612). Provider-specific size/mime limits |
| List `limit` | Clamped to `[1, 100]` (default 20) |
| List `offset` | Min 0 (default 0) |

## Error Reference

See [Error Codes](/api/errors) for the full table — consultation namespace is
`CN-16xx`. Upstream provider failures (`CN-1606`, `CN-1607`, `CN-1608`) map to
**502 Bad Gateway**, not 500, so monitoring can distinguish Aarokya-side breakage
from Narayana-side breakage.
